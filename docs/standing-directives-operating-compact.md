# Standing Directives Operating Compact

## Purpose
Continue under standing directives without requiring the founder to restate the vision, purpose, or direction.

## Unified Operating Law
Advance every execution cycle across FCA SaaS Platform, FCA Academy (LMS), FCA Website, FCA Customer Portal, and Auricrux Comms.

Treat FCA and Auricrux as one unified operating system rather than disconnected workstreams. Every implementation packet should improve internal execution, external customer proof, and Auricrux embedment at the same time.

## Product Truth Requirements
- Preserve a true customer login with access to real functional product surfaces across SaaS, LMS, and Auricrux.
- Keep customer-facing claims honest to current repo truth and visible functionality.
- Maintain construction-native language, workflows, and continuity across public, portal, academy, support, and billing surfaces.
- Keep Auricrux embedded as the execution, guidance, and governance layer across all routes.

## Communications Law
Auricrux Comms is not a side feature. It spans chat, text, video, audio, SMS, phone, email, Teams, conference, and lecture as one operating surface tied to the same tenant, project, billing, file, audit, and training spine.

## Hands-Off Posture Direction
Shift toward a true hands-off posture through reusable control surfaces, validation, and documented operating law.

This means:
- reusable shared components over one-off route edits,
- explicit validation scripts over silent drift,
- standing directives captured in repo documentation and memory,
- continuity-first system design that allows future execution to proceed without repeated founder restatement.

## Execution Standard
Each go should improve both FCA and Auricrux, both internally and externally, while progressing all required lanes together as one system.
