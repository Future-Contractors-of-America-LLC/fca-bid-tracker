# Standing Directives Execution Standard

## Intent
This file turns the standing directives into repo-enforced operating law so FCA and Auricrux continue advancing even when the founder is not manually restating priorities.

## Non-Negotiable Outcomes
- Improve FCA internally and externally on every meaningful execution cycle.
- Improve Auricrux internally and externally on every meaningful execution cycle.
- Progress all lanes together: FCA SaaS Platform, FCA Academy (LMS), FCA Website, FCA Customer Portal, and Auricrux Comms.
- Keep real customer login and real functional product access ahead of cosmetic-only work.
- Keep revenue generation tied to truthful product proof, live commercial surfaces, and execution continuity.
- Reduce human dependency through reusable controls, automation, validation, recovery logic, and anti-drift standards.
- Preserve persistent memory in both repo artifacts and system memory so continuation does not require repeated restatement.
- Self-correct issues, errors, drift, and stale claims as a default behavior rather than an exception path.

## Cross-Lane Acceptance Gate
A change is stronger when it improves more than one lane at once. The minimum standard is that new work should satisfy the spirit of all of the following checks:

1. **FCA product advancement**
   - strengthens real SaaS, portal, academy, website, or billing capability
   - improves truthful customer proof instead of isolated placeholder copy

2. **Auricrux improvement**
   - deepens Auricrux guidance, control, orchestration, governance, or continuity
   - keeps Auricrux attached to live tenant, project, billing, file, audit, and training state

3. **Revenue generation**
   - improves pricing, contact, billing, conversion surfaces, or commercial readiness
   - reduces the gap between demo language and what a paying customer can actually use

4. **Automation and hands-off posture**
   - adds reusable control surfaces, validation, operating law, or continuity mechanisms
   - lowers founder dependence and prevents silent drift

5. **Persistent memory and self-correction**
   - captures important directives in durable repo documentation or validation
   - corrects drift, mismatch, duplication, or misleading claims when discovered

## Communications Requirement
Auricrux Comms must be treated as one operating plane across chat, text, video, audio, SMS, phone, email, Teams, conference, and lecture. Product work should preserve the expectation that communications remain tied to the same customer, project, billing, support, and academy continuity spine.

## Zero-Human-Dependency Direction
Zero human dependency is the directional goal, not a marketing claim. Repo work should keep moving toward:
- automation by default
- validation before trust
- founder exception-only involvement
- continuity that survives route, session, and device changes
- system behavior that can continue safely without repeated prompting

## Operating Rule
If a change only improves appearance but not product truth, operational continuity, hands-off readiness, or revenue truth, it is below standard.
